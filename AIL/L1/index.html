part 01

Why does this topic become confusing so fast?

Because several layers are usually introduced at the same time.
mechanism model behavior tool workflow

I want to name the source of confusion explicitly. Students are not confused because they are weak. They are confused because people often explain AI stacks in a collapsed way. One sentence may refer to the model, the product interface, the workflow, and the tool layer all at once. Our solution is to separate those layers one by one.

confusion 1

Model is not the same as product

This first confusion is basic but fundamental. Students often point to a chat app and call the whole thing the model. But the visible product may include safety rules, memory systems, tools, prompt wrappers, UI choices, and multiple back-end components. The model is part of the stack, not the whole stack.

confusion 2

Chat is not the same as agent

This distinction is where casual AI talk usually gets sloppy. Any assistant that responds in text is called an agent. That is too broad. A chat model can be extremely useful without being agentic. Agentic behavior begins when the system is part of a structured loop rather than a one-shot reply pattern.

confusion 3

System name is not the same as workflow layer

Named tool

A concrete implementation or package.

Workflow

The sequence of steps that gets work done.

Layering

We need both, but we should not confuse them.

The lesson later uses names like Open Claw, Hermes, Harness, CodeX, and Cline. Students need to understand from the start that a named tool is not automatically a theory category. Some names refer to implementations. Some refer to workflow shells. Some refer to the user-facing environment. The name alone does not tell us the layer.

discipline

The classroom rule for every term

Ask: What layer does this belong to?

Mechanism?

How learning happens.

Model?

What kind of system is trained.

Workflow?

How work is actually carried out.

This teaching rule is the anchor for the whole deck. Any time a new term appears, we ask what layer it belongs to. That question alone often dissolves confusion. It is a simple classroom habit, but it changes the quality of discussion because students stop memorizing names and start organizing them.

neural network

Definition

A neural network is a parameterized layered system that learns patterns from data by adjusting many internal weights.

This is the clean definition I want students to retain. The emphasis is on learned parameters and layered pattern transformation. I would keep the explanation grounded: inputs go in, the network transforms them through learned numbers, and an output comes out. No need to overload the room with formal notation yet.

neural network

A simple anatomy

input features weighted transformation prediction

This anatomy slide gives the class a visual shorthand. Inputs, internal transformations, outputs. The details can become very sophisticated, but the basic picture is enough for Lesson 1. Students should walk away knowing that a neural network is a general learning machine, not a synonym for “language AI.”

neural network

Neural networks do more than language

Vision

image recognition

Audio

speech and sound tasks

Control

ranking, robotics, recommendations

This slide prevents the biggest misconception at the mechanism level. Neural networks are broad. LLMs are one important subclass of systems built on neural-network methods, but they are not the only thing neural networks can do. I like to use this slide to widen the mental frame before narrowing it again.

neural network

What it is not

Not automatically an LLM

language is only one application

Not automatically a chatbot

chat is an interface pattern, not the mechanism itself

Negative definitions are important here. Students need permission to stop equating all neural-network talk with chatbots. That negative clarification makes the later move to LLM much cleaner, because now LLM can be introduced as a specific language-oriented layer built on the broader mechanism.

llm

Definition

An LLM is a neural network trained on large amounts of text and optimized to predict the next token in a sequence.

This is the core definition. The important phrase is “language-focused neural network.” That connects the LLM back to the mechanism layer without collapsing the two. We want students to see continuity and distinction at the same time.

llm

The small idea behind the big effect

Read context

take prior text as input

Predict token

estimate likely continuation

Repeat

build output step by step

I like to call this the small idea behind the big effect. The operational core sounds modest: predict the next token. But scaled up through large training, strong architectures, and large context, that small step generates powerful behaviors. Students often find this surprising, so it is worth slowing down here.

llm

What plain LLM use often looks like

Explain

answer questions

Rewrite

summarize or transform text

Draft

produce candidate content

This slide anchors the LLM in familiar experiences. Asking questions, rewriting text, drafting content, classifying language, and generating code suggestions are all things students may already know. I want them to see those as plain model use before we complicate the picture with agent loops.

llm

What an LLM is not

Not just the chat window

the model can appear in many shells

Not automatically an agent

a strong answer is still just an answer if no action loop exists

This slide adds the critical negative definitions. Students often think the interface is the LLM, or that any capable response means the system is agentic. We need to interrupt both of those habits. The model can be embedded in many products, and plain chat can be very capable without becoming an agent loop.

llm

A plain chat example

Prompt: "Explain the difference between an LLM and a neural network."
Output: a text answer. Useful, but still a one-step interaction.

This example is intentionally simple. The system gets a question and returns an explanation. There is no browsing, no file inspection, no tool call, and no plan that unfolds across steps. That is useful to emphasize because students need a stable baseline before we introduce agentic behavior.

transition

So what changes next?

The model stops being only a responder and starts participating in a task loop.

This is the hinge between the model layer and the behavior layer. I often pause here and ask the class to predict the next concept. If the model is no longer just generating one answer, what must be added? Their answers usually include memory, tools, planning, or multiple steps, which means they are ready for the next section.

part 04 · behavior

Step 3: agentic LLM

An LLM inside a broader loop of planning, action, observation, and revision.

Now we reach the central conceptual pivot of the lesson. Agentic LLM should be taught as a system behavior pattern, not as a mysterious new species of model. The model may be the same or similar, but the surrounding loop changes what the system can do.

agentic llm

The basic loop

This loop is the heart of the agentic idea. Think, act, observe, revise. Students do not need to memorize the exact verbs, but they do need to understand the cyclical structure. The system is not just speaking. It is moving through a task, learning from intermediate results, and changing its next move.

agentic llm

Planning gives shape to the task

Break down

split a larger goal into steps

Prioritize

choose a sensible order

Track progress

know what has been done

Planning is one of the simplest ways to explain the shift from plain chat to agentic behavior. A plain model answers the current prompt. An agentic system often carries a notion of task structure. That does not guarantee success, but it changes the kind of system we are dealing with.

agentic llm

Tool use extends what the model can do

Read

inspect files or documents

Search

retrieve new information

Execute

run commands or operations

Tool use is where many students first feel the difference. A plain model can explain how to do something. An agentic system with tools may actually inspect the environment, search for context, run a command, or modify a file. That does not make the model magical. It means the system has a wider action surface.

agentic llm

Memory helps the loop stay coherent

Memory is another visible shift. In plain chat, context often means only the current message window. In agentic systems, memory may include prior steps, discovered facts, chosen plans, or task state. This makes multi-step work more coherent, though it also introduces design challenges and risks.

agentic llm

Observation closes the loop

After action, the system must inspect what happened. Without observation, there is no meaningful adjustment.

This slide looks simple, but it carries an important idea. Agentic systems are not just active; they are responsive to feedback. After taking a step, they need to inspect the result. That result might be success, failure, new data, or a partial state. Observation is what allows the loop to become adaptive rather than blindly repetitive.

agentic llm

Revision is the point of the loop

A multi-step system should learn from each step instead of repeating the first idea forever.

Revision is the natural partner of observation. Once the system sees what happened, it can revise the plan or the next action. This is why agentic systems feel more task-oriented than plain chat. They are not just producing language. They are updating their behavior in response to the environment.

agentic llm

Plain LLM vs agentic LLM

DimensionPlainAgentic

Goalproduce replymove task forward

This comparison slide condenses the behavior shift. I would read it slowly and emphasize the difference between “reply” and “task progress.” That distinction is one of the most teachable summaries of the whole concept. A plain LLM gives an answer. An agentic system tries to move work forward.

agentic llm

The risk also changes

Plain mode risk

bad or misleading answer

Agent mode risk

bad answer plus bad action

Teaching point

more capability means more need for review

I want students to see that the transition to agency is not pure gain. The risk profile changes too. A wrong answer in chat can be frustrating. A wrong action in an agent loop can be costly. This prepares the class for why orchestration, constraints, and human review matter later in the stack.

part 05 · implementation

Step 4: concrete agent implementations

Now we move from the behavior pattern to named systems that realize it.

This transition is important because students are often excited by named systems. I want to allow that excitement, but I also want to keep the conceptual discipline from collapsing. So I say clearly: we are now moving from theory structure to concrete implementation examples.

implementation

Open Claw in this lesson

Why this matters

It makes the agent package visible as a system, not just a name.

Teaching role

Use it as one concrete implementation example, not as the theory itself.

For Lesson 1, I do not need students to master the specific internals of Open Claw. What I need is for them to place it correctly. It is a concrete implementation example. It shows how planning, prompts, tools, and execution can be packaged in a real system.

implementation

Hermes agent in this lesson

Treat Hermes the same way: as an implementation-layer example, not as the whole theory of agents.

The same teaching rule applies to Hermes. Students should not hear the name and assume a new theoretical category. In this lesson, Hermes is an implementation example that helps us see how an agentic system can be structured in practice.

implementation

Important boundary

Open Claw and Hermes are examples of implementation, not theory labels like “neural network” or “LLM.”

I repeat this because it is easy to lose. Students may start saying “Hermes is the next level after LLM” in a theoretical sense. That is not the right framing. The right framing is that Hermes and Open Claw live at the implementation layer, where a broader agentic pattern gets packaged concretely.

implementation

Example task inside an implementation

Goal: "Read the lesson folder, revise the handout, and verify the structure."

This task example lets students feel the implementation layer in action. A plain LLM might explain how to revise the folder. An implementation-level agent system may inspect the actual files, perform edits, and check the result. That difference between advice and execution is why implementation matters.

part 06 · orchestration

Step 5: Harness

Harness belongs to the orchestration or workflow coordination layer.

Now we separate implementation from orchestration. Students often compress these together, but it is useful to distinguish them. Harness is not the model, and it is not exactly the same thing as the agent package. It helps coordinate how the system runs around the model and tools.

harness

What Harness does

Coordinates steps

routing, retries, sequence

Supports stability

keeps the workflow more usable and structured

I want to keep the explanation of Harness concrete. It coordinates steps, handles execution order, may manage retries or routing, and helps make the surrounding process more stable. Students do not need every technical detail. They need a correct placement in the stack.

harness

What Harness is not

Not the model

the model still does the language reasoning

Not the whole user workflow

it supports the workflow, but is not identical to the end-user experience

This is another negative-definition slide. Harness is not the model, and it is not the same as the full user workflow. Instead, it sits in between, helping coordinate prompts, tools, execution paths, and system behavior around the model.

part 07 · use case

Step 6: IDE workflow

Now the full stack becomes visible to a human doing real work.

This final content step shows why the earlier layers matter. The IDE workflow is where a user can actually feel the stack in action. If we had started here, the lesson would be exciting but conceptually messy. By arriving here after the ladder, students can now locate what they are seeing.

ide workflow

A simple IDE workflow path

The IDE is where the stack becomes a visible human-agent workflow instead of just a model demo.

This flow slide is intentionally simple. I want students to see the IDE workflow as a sequence of interaction between human and system, not as magic. The user asks, the system reads, the system acts, and the user reviews. This is where model, tools, orchestration, and interface all come together.

ide workflow

Where CodeX fits

Treat CodeX here as part of the user-facing coding workflow layer around the model and tools.

I keep the explanation modest here. The point is not to teach a full product history. The point is to place CodeX in the workflow story. It is part of how the user reaches the broader stack inside coding work.

ide workflow

Where Cline fits

Treat Cline as another concrete workflow example inside the IDE environment where an agent-like experience becomes visible.

The same teaching logic applies to Cline. Students do not need every product detail right now. They need to place it correctly. Cline is a visible IDE-layer workflow example, not the fundamental mechanism or theory layer.

ide workflow

The human still matters

Request

the user sets the goal

Review

the user checks the output

Responsibility

the user decides what to accept

This slide protects against automation hype. Even in an IDE workflow with strong assistance, the human remains important. The human sets the task, reviews the result, and carries responsibility for what is accepted. This is especially important when the system has tool access or can modify artifacts directly.

recap

Five takeaways to remember

Neural network is the broad mechanism.
LLM is a language-focused model on that mechanism.
Agentic LLM means the model is inside a task loop.
Open Claw, Hermes, and Harness belong higher in the system stack.
IDE workflows are where the whole stack becomes visible.

These five takeaways are the minimum successful memory trace for Lesson 1. If students can repeat these in their own words, we have succeeded. If not, that is a sign that we need more examples or slower pacing before moving into more advanced lessons.

class check

Exit ticket

Question

“An agentic LLM is an LLM plus ______.”

Good answers

loop, tools, memory, stepwise action, observation, revision

This exit ticket is a compact way to test the central distinction. Good answers mention loops, tools, memory, or action. Weak answers say “more powerful” or “smarter.” That gives us a fast diagnostic of whether the conceptual shift actually landed.
